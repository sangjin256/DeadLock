# DeadLock 게임 디자인

## 문서 목적

이 문서는 DeadLock 퍼즐 게임 부활과 모바일 이식 작업의 로컬 디자인 기준이다.

## 현재 핵심

- 플레이어는 프로세스의 색상 슬롯을 호환되는 리소스 노드에 연결한다.
- 플레이어가 필요한 연결을 모두 예약한 뒤 라운드 단위 시뮬레이션이 실행된다.
- 리소스는 수용량, 색상, 동시 연결, 색상 전환, 빈 색상, 시계 제약을 가질 수 있다.
- 성공 조건은 레벨 규칙 안에서 모든 프로세스 노드가 완료되는 것이다.

## 새로 추가할 퍼즐: Relay

Relay는 새 리소스 종류가 아니라, 기존 리소스 위에 추가로 얹히는 공간 퍼즐 모디파이어다. Basic, Capacity, Simultaneous, ColorSwitch, EmptyColor, Clock 같은 리소스가 먼저 있고, 그 리소스 두 개가 Relay 관계로 묶일 수 있다.

Relay의 목적은 보드 위에서 멀리 떨어진 리소스들이 서로 영향을 주게 만들어, 단순히 가까운 리소스에 선을 연결하는 문제를 넘어 공간적 판단을 요구하는 것이다. 색을 새 규칙으로 더 많이 추가하면 플레이어가 헷갈릴 수 있으므로, Relay는 색보다 라인, 끝 장식, 상태 변화, 모션으로 구분한다.

공통 규칙:

- Relay는 항상 두 리소스를 한 쌍으로 묶는다.
- Relay 관계는 프로세스의 필요 색과 별개의 규칙이다.
- Relay가 붙어도 각 리소스의 기존 규칙은 유지된다.
- 기본 상태에서는 두 리소스가 Relay로 묶여 있다는 힌트만 보여주고, 리소스 선택 또는 hover/tap 시 전체 연결선을 보여준다.
- 모바일에서는 hover가 없으므로, Relay가 붙은 리소스를 탭했을 때 연결된 상대 리소스와 전체 Relay 라인이 잠시 강조된다.

### Relay Link

Relay Link는 연결된 두 리소스가 서로 같은 상태 그룹에 묶이는 퍼즐이다.

기본 아이디어는 "한쪽을 사용하고 있으면 다른쪽도 영향을 받는다"이다. 예를 들어 한쪽 리소스가 이번 라운드에 사용 중이면, Link로 묶인 다른 리소스도 잠김 상태가 되거나 같은 사용 상태로 표시될 수 있다. 이 규칙은 플레이어가 리소스를 각각 독립된 노드로만 보지 않고, 떨어진 두 리소스를 하나의 세트로 보게 만든다.

가능한 규칙 방향:

- 공유 사용: 두 리소스 중 하나가 사용되면 다른쪽도 이번 라운드에 사용된 것으로 취급한다.
- 상호 잠금: 한쪽을 선택하면 다른쪽은 같은 라운드에 선택할 수 없다.
- 공유 쿨다운: 한쪽을 사용하면 둘 다 다음 라운드까지 대기 상태가 된다.

초기 구현 후보는 `상호 잠금`이 가장 직관적이다. 한쪽이 켜지면 다른쪽이 잠기는 식이라 시각적으로도 읽기 쉽고, 레벨 설계에서도 "둘 중 어느 쪽을 이번 라운드에 쓸 것인가"라는 선택을 만들 수 있다.

### Relay Transfer

Relay Transfer는 한쪽 리소스의 입력 또는 상태가 다른쪽 리소스로 전달되는 퍼즐이다.

기본 아이디어는 "A에 연결한 결과가 B에도 영향을 준다"이다. Link가 두 리소스를 같은 묶음으로 보는 규칙이라면, Transfer는 방향성이 있는 전달 규칙이다. 따라서 Transfer는 Link보다 방향 표시가 중요하다.

가능한 규칙 방향:

- 색 전달: 송신 리소스에 들어온 색이 수신 리소스의 임시 색 또는 활성 색으로 전달된다.
- 상태 전달: 송신 리소스가 활성화되면 수신 리소스도 활성 조건 일부가 충족된다.
- 입력 복사: 송신 리소스에 연결된 입력이 수신 리소스에도 복사되어, 한 번의 연결이 두 위치에 영향을 준다.

초기 구현 후보는 `상태 전달`이 가장 안전하다. 색 전달은 ColorSwitch, EmptyColor와 혼동될 수 있고, 입력 복사는 규칙 설명이 길어질 수 있다. 상태 전달은 "이쪽을 켜면 저쪽이 열린다"에 가까워 모바일에서도 이해시키기 쉽다.

### Relay를 리소스 타입으로 만들지 않는 이유

Relay는 리소스 하나의 성질이 아니라 두 리소스 사이의 관계다. 그래서 `RelayResource`처럼 독립 리소스로 만들기보다, 레벨 데이터에서 두 리소스 사이에 `RelayLink` 또는 `RelayTransfer` 관계를 추가하는 방식이 적합하다.

이렇게 하면 같은 Basic 리소스라도 어떤 레벨에서는 일반 Basic이고, 다른 레벨에서는 Relay Link가 붙은 Basic이 될 수 있다. 기존 퍼즐 규칙을 버리지 않고 조합을 늘릴 수 있다는 장점이 있다.

### 레벨 디자인 활용

Relay Link는 "둘 중 하나만 쓸 수 있는 선택"을 만드는 챕터에 적합하다. 플레이어는 단순히 색이 맞는 리소스를 찾는 것이 아니라, 그 리소스를 쓰면 멀리 있는 다른 리소스가 막히는지까지 고려해야 한다.

Relay Transfer는 "먼 곳의 리소스를 간접적으로 작동시키는 흐름"을 만드는 챕터에 적합하다. 직접 연결하기 어려운 리소스를 다른 리소스를 통해 열거나, 특정 순서로 활성화해야 하는 퍼즐에 사용할 수 있다.

두 규칙은 같은 챕터에 바로 섞기보다, 각각 별도의 퍼즐 챕터로 소개하는 것을 우선한다. Link는 공유/잠금의 개념을 먼저 학습시키고, Transfer는 방향성과 전달 개념을 별도로 학습시킨다.

## 디자인 우선순위

1. 원작 퍼즐의 가독성과 논리를 보존한다.
2. 모바일 조작성을 개선한다.
3. 게임 상태를 흐리지 않는 범위에서 비주얼 피드백을 강화한다.
4. 퍼즐 규칙은 Unity 밖에서도 테스트 가능하게 만든다.

## 열린 디자인 메모

- 현재 원/사각형 노드 비주얼은 변경될 수 있다.
- 연결은 물방울, 젤, 표면장력 시각 언어를 탐색한다.
- 정적 레벨 ScriptableObject는 유지할 수 있지만, 런타임 규칙은 순수 Domain 객체로 이동하는 것을 목표로 한다.
